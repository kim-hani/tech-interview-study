# PCB (Process Control Block)

---

## 1. PCB란?

**PCB(Process Control Block)** 는 운영체제가 프로세스를 관리하기 위해 사용하는 **프로세스 관리 정보 저장 구조체**이다.

프로세스는 실행 중인 프로그램을 의미한다.  
운영체제는 여러 프로세스를 동시에 관리해야 하므로, 각 프로세스의 상태와 실행 정보를 따로 저장해 둘 필요가 있다.

이때 운영체제가 각 프로세스마다 생성하여 관리하는 정보 블록이 PCB이다.

```text
프로세스 A → PCB A
프로세스 B → PCB B
프로세스 C → PCB C
```

즉, PCB는 운영체제가 특정 프로세스를 식별하고, 실행 상태를 저장하고, 다시 실행할 수 있도록 관리하는 자료구조이다.

프로세스가 생성되면 운영체제는 해당 프로세스에 대한 PCB를 생성하고, 프로세스가 종료되면 PCB도 함께 제거한다.

이때 PCB는 사용자 영역이 아니라 **커널 영역에서 관리**된다.

PCB는 운영체제가 관리하는 중요한 자료구조이다.  

따라서 일반 사용자 프로그램이 직접 PCB를 수정할 수 없다.

---

## 2. PCB가 필요한 이유

운영체제는 한 번에 하나의 프로세스만 관리하는 것이 아니라, 여러 프로세스를 번갈아 실행한다.

예를 들어 다음과 같은 프로그램들이 동시에 실행 중이라고 하자.

```text
브라우저
음악 플레이어
IDE
터미널
백그라운드 서비스
```

CPU는 매우 빠르게 여러 프로세스를 번갈아 실행하기 때문에 사용자는 여러 프로그램이 동시에 실행되는 것처럼 느낀다.

하지만 CPU가 어떤 프로세스를 실행하다가 다른 프로세스로 전환하려면, 기존 프로세스가 어디까지 실행되었는지 저장해야 한다.

예를 들어 프로세스 A가 실행 중이었다고 하자.

```text
프로세스 A 실행 중
  ↓
프로세스 B로 전환 필요
  ↓
프로세스 A의 현재 실행 상태 저장
  ↓
프로세스 B의 실행 상태 복원
  ↓
프로세스 B 실행
```

이때 프로세스 A의 실행 상태를 저장하는 곳이 PCB이다.

따라서 PCB가 없다면 운영체제는 프로세스를 중단했다가 다시 이어서 실행할 수 없다.

---

## 3. PCB에 저장되는 정보

PCB에는 프로세스를 관리하기 위한 다양한 정보가 저장된다.

대표적인 정보는 다음과 같다.

| 정보 | 설명 |
|------|------|
| 프로세스 ID | 프로세스를 식별하기 위한 고유 번호 |
| 프로세스 상태 | 준비, 실행, 대기, 종료 등 현재 상태 |
| 프로그램 카운터 | 다음에 실행할 명령어의 주소 |
| CPU 레지스터 값 | 문맥 교환 시 복원해야 하는 레지스터 값 |
| CPU 스케줄링 정보 | 우선순위, 스케줄링 큐 포인터 등 |
| 메모리 관리 정보 | 페이지 테이블, 세그먼트 테이블 등 메모리 관련 정보 |
| 입출력 상태 정보 | 열린 파일, 사용 중인 입출력장치 정보 |
| 계정 정보 | CPU 사용 시간, 사용자 ID, 프로세스 실행 시간 등 |

PCB에 저장되는 정보는 운영체제마다 다를 수 있다.  
하지만 핵심 목적은 같다.

```text
프로세스를 식별하고
프로세스의 실행 상태를 저장하며
필요할 때 다시 실행할 수 있도록 관리한다.
```

---

## 4. 프로세스 상태와 PCB

PCB에는 프로세스의 현재 상태가 저장된다.

프로세스 상태는 운영체제가 프로세스를 어떻게 관리할지 결정하는 데 사용된다.

대표적인 프로세스 상태는 다음과 같다.

| 상태 | 설명 |
|------|------|
| 생성 상태 | 프로세스가 생성 중인 상태 |
| 준비 상태 | CPU를 할당받기 위해 기다리는 상태 |
| 실행 상태 | CPU를 할당받아 실행 중인 상태 |
| 대기 상태 | 입출력 작업 등 특정 사건을 기다리는 상태 |
| 종료 상태 | 프로세스 실행이 끝난 상태 |

프로세스 상태 변화는 다음과 같이 이해할 수 있다.

```text
생성
 ↓
준비
 ↓
실행
 ↓
종료
```

입출력 작업이 필요한 경우에는 실행 상태에서 대기 상태로 이동할 수 있다.

```text
실행 상태
  ↓ 입출력 요청
대기 상태
  ↓ 입출력 완료
준비 상태
```

운영체제는 PCB에 저장된 프로세스 상태를 확인하여 어떤 프로세스를 실행할지, 어떤 프로세스를 기다리게 할지 결정한다.

## 5. 문맥 교환과 PCB

**문맥 교환(Context Switching)** 은 CPU가 실행 중인 프로세스를 다른 프로세스로 바꾸는 작업이다.

여기서 **문맥(Context)** 은 프로세스를 다시 실행하기 위해 필요한 실행 상태 정보를 의미한다.

문맥에는 다음과 같은 정보가 포함된다.

```text
프로그램 카운터 값
레지스터 값
스택 포인터 값
프로세스 상태
메모리 관련 정보
```

이 정보들이 PCB에 저장된다.

문맥 교환 과정은 다음과 같다.

```text
1. 프로세스 A가 실행 중이다.
2. 인터럽트, 입출력 요청, 타임 슬라이스 만료 등으로 프로세스 전환이 필요해진다.
3. 운영체제는 프로세스 A의 현재 실행 상태를 PCB A에 저장한다.
4. 운영체제는 다음에 실행할 프로세스 B를 선택한다.
5. PCB B에 저장된 실행 상태를 CPU로 복원한다.
6. CPU는 프로세스 B를 이어서 실행한다.
```

그림으로 표현하면 다음과 같다.

```text
CPU 실행 중
┌──────────────┐
│ 프로세스 A   │
└──────┬───────┘
       ↓ 현재 상태 저장
┌──────────────┐
│ PCB A         │
└──────────────┘

┌──────────────┐
│ PCB B         │
└──────┬───────┘
       ↓ 실행 상태 복원
┌──────────────┐
│ 프로세스 B   │
└──────────────┘
```

문맥 교환은 멀티태스킹 운영체제에서 필수적이다.

다만 문맥 교환 중에는 실제 사용자 프로그램의 작업이 수행되는 것이 아니라, 운영체제가 상태를 저장하고 복원하는 작업을 수행한다.  
따라서 문맥 교환이 너무 자주 발생하면 성능 저하가 발생할 수 있다.

## 6. PCB와 CPU 레지스터의 관계

PCB는 CPU 레지스터와 밀접한 관련이 있다.

프로세스가 실행 중일 때는 프로그램 카운터, 스택 포인터, 범용 레지스터, 플래그 레지스터 등의 값이 CPU 내부에 존재한다.

하지만 CPU가 다른 프로세스를 실행해야 한다면, 현재 CPU 레지스터 값들을 보존해야 한다.

예를 들어 프로세스 A가 실행 중일 때 CPU 내부 상태가 다음과 같다고 하자.

```text
PC = 1000
SP = 8000
R1 = 10
R2 = 20
Flag = 0
```

프로세스 A가 중단되고 프로세스 B가 실행되어야 한다면, 운영체제는 이 값들을 PCB A에 저장한다.

```text
PCB A
├── PC = 1000
├── SP = 8000
├── R1 = 10
├── R2 = 20
└── Flag = 0
```

나중에 프로세스 A를 다시 실행할 때 운영체제는 PCB A에 저장된 값을 CPU 레지스터에 복원한다.

```text
PCB A의 값 복원
  ↓
CPU 레지스터 복구
  ↓
프로세스 A 실행 재개
```

즉, PCB는 프로세스의 실행 흐름을 멈췄다가 다시 이어서 실행할 수 있게 해주는 핵심 정보 저장소이다.

## 7. PCB와 스케줄링

운영체제는 CPU를 어떤 프로세스에게 할당할지 결정해야 한다.

이 작업을 **CPU 스케줄링**이라고 한다.

CPU 스케줄링을 위해 운영체제는 PCB에 저장된 정보를 활용한다.

예를 들어 PCB에는 다음과 같은 스케줄링 정보가 저장될 수 있다.

```text
프로세스 우선순위
프로세스 상태
CPU 사용 시간
대기 시간
스케줄링 큐 정보
```

운영체제는 이 정보를 바탕으로 다음에 실행할 프로세스를 선택한다.

```text
준비 큐에 있는 프로세스들
  ↓
스케줄러가 PCB 정보 확인
  ↓
실행할 프로세스 선택
  ↓
CPU 할당
```

예를 들어 우선순위 기반 스케줄링에서는 우선순위가 높은 프로세스가 먼저 CPU를 할당받을 수 있다.

## 8. PCB와 프로세스, 스레드

PCB는 기본적으로 **프로세스 단위**의 정보를 저장한다.

프로세스는 실행 중인 프로그램이며, 독립적인 메모리 공간과 자원을 가진다.

반면 스레드는 프로세스 안에서 실행되는 작업 흐름이다.

```text
프로세스
├── 스레드 1
├── 스레드 2
└── 스레드 3
```

프로세스 단위의 정보는 PCB에 저장된다.

```text
PCB
├── 프로세스 ID
├── 프로세스 상태
├── 메모리 관리 정보
├── 파일 정보
└── 자원 정보
```

스레드 단위의 실행 정보는 일반적으로 **TCB(Thread Control Block)** 에 저장된다.

```text
TCB
├── 스레드 ID
├── 스레드 상태
├── 프로그램 카운터
├── 레지스터 값
└── 스택 정보
```

정리하면 다음과 같다.

| 구분 | PCB | TCB |
|------|-----|-----|
| 의미 | Process Control Block | Thread Control Block |
| 관리 대상 | 프로세스 | 스레드 |
| 주요 정보 | 프로세스 ID, 상태, 메모리 정보, 열린 파일 정보 | 스레드 ID, 상태, 레지스터 값, 스택 정보 |
| 사용 목적 | 프로세스 관리 | 스레드 관리 |

운영체제마다 내부 구조는 다를 수 있지만, 개념적으로 PCB는 프로세스 관리 정보, TCB는 스레드 관리 정보라고 이해하면 된다.


---

## 11. 정리

- PCB는 **Process Control Block**의 약자이다.
- PCB는 운영체제가 프로세스를 관리하기 위해 사용하는 정보 저장 구조체이다.
- 프로세스가 생성되면 운영체제는 해당 프로세스에 대한 PCB를 생성한다.
- 프로세스가 종료되면 PCB도 함께 제거된다.
- PCB에는 프로세스 ID, 프로세스 상태, 프로그램 카운터, CPU 레지스터 값, 메모리 관리 정보, 입출력 상태 정보 등이 저장된다.
- PCB는 문맥 교환에서 핵심적인 역할을 한다.
- 문맥 교환 시 운영체제는 현재 실행 중인 프로세스의 실행 상태를 PCB에 저장하고, 다음 프로세스의 PCB에서 실행 상태를 복원한다.
- PCB는 프로세스 자체가 아니라 프로세스를 관리하기 위한 정보이다.
- PCB는 일반적으로 커널 영역에서 관리된다.
- Java 프로그램도 운영체제 관점에서는 하나의 프로세스로 실행되며, 해당 프로세스를 관리하기 위해 PCB가 생성된다.
- 스레드 단위의 실행 정보는 개념적으로 TCB(Thread Control Block)와 관련된다.
- 문맥 교환은 멀티태스킹에 필수적이지만, 지나치게 자주 발생하면 성능 저하가 발생할 수 있다.
